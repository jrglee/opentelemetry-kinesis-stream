# Encoding × codec performance

Empirical numbers for the two live encodings (`otlp_proto`, `otlp_json`) across
the seven codecs the exporter exposes
(`none`/`gzip`/`zstd`/`snappy`/`x-snappy-framed`/`zlib`/`deflate`), at
four dataset profiles and six batch sizes. Read alongside the tables — single
numbers without their (profile, codec, batch size) context are noise.

```
make perf | tee benchmark-raw.txt
```

Inputs are seeded (`PerfSeed = 0xC0FFEEBABE`) so `B/rec`, `total`, and `ratio`
are byte-identical on any host. `p50`/`p90` vary with CPU; ordering between
rows on the same host is the load-bearing comparison.

## Environment

- Apple M2 Ultra, darwin/arm64, 24 cores
- Go 1.26.4, klauspost/compress (pinned)
- Captured at the `make perf-parallel` defaults: `-benchtime=2s -count=3`,
  four phases concurrent at `GOMAXPROCS=6`.

## OTel-Arrow was considered, benchmarked, and removed

An `otel_arrow` encoding shipped briefly under
[ADR-0018](docs/adr/0018-implement-otel-arrow-encoding.md): a fresh
`arrow_record.Producer` per Marshal, a fresh `Consumer` per Unmarshal, so
that each Kinesis record decoded in isolation despite OTAP being designed
as a stateful gRPC stream. It was benchmarked on this same harness and
removed per [ADR-0020](docs/adr/0020-remove-otel-arrow-encoding.md). The
numbers behind the decision, in three lines:

- **Size win, on the right shape, at scale.** On `metrics-high-frequency`
  zstd the Arrow `B/rec` crossed below `otlp_proto` at n=1,000 and kept
  falling: 8.7 B vs proto 13.7 B (0.64×) at n=1k, 4.8 B vs 11.9 B (0.40×)
  at n=100k. The compression model is real.
- **Paid for in CPU.** At n=10k zstd Arrow encoded in p50 ≈ 15.6 ms vs
  proto's 2.58 ms (6×); at n=100k, 209 ms vs 23 ms (9×). The win required
  microbatches of ≥ 1,000 datapoints AND CPU to spare.
- **Unsafe on high-cardinality.** On the `metrics-high-cardinality`
  profile Arrow showed no size advantage at any batch size, and the
  upstream `arrow_record.Producer` v0.48.0 panicked with **"Too many
  consecutive schema updates"** at n=100,000. The per-record self-contained
  IPC variant cannot amortize the dictionary across records, which is
  exactly the case where OTAP's cross-batch dictionary delta would have
  paid off — and that delta only exists in a long-lived gRPC stream.

ADR-0020 reads the rest of the verdict: Kinesis's store-and-forward,
per-record delivery forfeits the statefulness that motivates OTAP, so the
self-contained variant pays Arrow's schema overhead on every record without
the payoff. The tables below were regenerated after removal and cover only
the encodings that ship.

## Conclusions

Read alongside the tables above; single-row claims out of context lie.

### Pick `otlp_proto` + zstd as the default

Across every profile and batch size, `otlp_proto` + `zstd` is the fastest
encode/decode pair and the cheapest in allocations. On
`metrics-high-frequency` its `B/rec` plateaus around 12 B at n ≥ 1,000
(p50 ≈ 2.54 ms at n=10k). `otlp_json` wins ~30–40 % on `B/rec` at the same
cell but costs 2–3× the encode latency and an order of magnitude more
allocations. Use proto + zstd unless a specific reason pushes you
elsewhere.

### `otlp_json` is genuinely competitive on size

JSON's verbosity is highly redundant, so zstd hits ~22–24× ratios at
n ≥ 1,000 on the high-frequency and balanced profiles. At n=100k zstd:

- `metrics-high-frequency`: JSON 9.30 B/rec vs proto 12 B/rec (0.78×)
- `metrics-balanced`: JSON 13 B/rec vs proto 15 B/rec (0.87×)

Reach for it when shard bandwidth is the binding constraint or downstream
consumers prefer human-readable records.

### Codec ranking

Within an encoding, codec choice has a clear ranking that holds across
profiles and batch sizes:

1. **`zstd`** — best ratio per CPU cycle. Default.
2. **`snappy` / `x-snappy-framed`** — half the ratio of `zstd` but 2–3×
   faster encode. Under CPU pressure.
3. **`gzip` / `zlib` / `deflate`** — equivalent ratios to each other,
   slightly better than `zstd` in a few cells, but 3–6× slower than `zstd`.
   Only for downstream compatibility.
4. **`none`** — only at n=1, where compression overhead exceeds the bytes
   saved.

### Picking a batch size

`B/rec` falls steeply from n=1 to n=1,000 and plateaus thereafter for both
encodings. **n ≈ 1,000 is the knee** — diminishing returns on batching
beyond that, with steeply rising encode wall time.

### Pair with the Kinesis record ceiling

`total` tells you when a configuration would exceed a stream's per-record
limit. With `max_record_size: 1 MiB` (default), at high-frequency zstd
both `otlp_proto` and `otlp_json` batch safely to n=100,000 (~1 MiB
record). With the 10 MiB opt-in ceiling the size question stops being
"will it fit" and becomes "what CPU and tail latency am I willing to
spend on each record."
