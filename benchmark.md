# Encoding × codec performance

Empirical numbers backing the architectural decision in
[docs/adr/0018-implement-otel-arrow-encoding.md](docs/adr/0018-implement-otel-arrow-encoding.md).

```
make perf | tee benchmark-raw.txt
```

Inputs are seeded (`PerfSeed = 0xC0FFEEBABE`) so `B/rec`, `total`, and `ratio`
are byte-identical on any host. `p50`/`p90` vary with CPU; ordering between
rows on the same host is the load-bearing comparison.

## Environment

- Apple M2 Ultra, darwin/arm64, 24 cores
- Go 1.26.4, `otel-arrow` v0.48.0, klauspost/compress (pinned)
- Captured at `-benchtime=500ms -count=1`. `make perf` defaults to
  `-benchtime=2s -count=3` for benchstat-quality numbers.

## Profiles

| Profile | Shape |
|---|---|
| `metrics-high-cardinality` | unique `request_id`/`user_id` per datapoint |
| `metrics-high-frequency` | small label fan-out, many datapoints per series |
| `metrics-balanced` | moderate of both |
| `traces-typical` | random trace/span IDs, fixed endpoint vocabulary |

`n ∈ {1, 10, 100, 1000, 10000, 100000}`. 1M was tried and cut — Arrow at 1M
takes ~3 s/call, multi-hour CI runtime, no new conclusions.

## Arrow producer limit

Upstream `arrow_record.Producer` v0.48.0 panics with **"Too many consecutive
schema updates."** on inputs where every datapoint introduces a new
dictionary entry — i.e. the high-cardinality profile at n ≥ 1000. The
harness recovers and marks the cell **—**. Cells skipped this way are a
real finding: Arrow is unsafe for genuinely high-cardinality metrics on
this transport, before any size or CPU consideration applies.

## Conclusions

Backed by the cells above; read alongside the tables, not in isolation.

### Pick OTLP-proto + zstd as the default

Across every profile and batch size, OTLP-proto + zstd is the fastest
encode/decode and the cheapest in allocations. On the high-frequency profile
its `B/rec` plateaus around 12 B at n ≥ 1000 (p50 ≈ 2.6 ms at n=10k). It
loses 30–60% on `B/rec` against the other two at large n, but the wall-time
gap is multiplicative: Arrow is 6–10× slower encode, JSON is 2–3×.

### OTLP-JSON is genuinely competitive on size

JSON's verbosity is highly redundant, so zstd hits 22–24× ratios at n ≥ 1000
in high-frequency and balanced profiles. At n=100k zstd:

- balanced: JSON 12.6 B/rec vs proto 15.4 vs Arrow 4.78
- high-cardinality: JSON 31.97 B/rec vs proto 37.47 (Arrow skipped)

Costs 2–3× the encode latency and an order of magnitude more allocations.
Reasonable choice when shard bandwidth is the binding constraint or
downstream consumers want human-readable.

### Arrow's size win is real — at scale, on the right shape

On `high-frequency` and `balanced` zstd, Arrow's `B/rec` falls below proto
at **n = 1,000** and the gap widens with n:

| profile | n | proto B/rec | Arrow B/rec | Arrow ÷ proto |
|---|---:|---:|---:|---:|
| high-frequency | 1,000 | 13.69 | 8.72 | 0.64× |
| high-frequency | 10,000 | 12.08 | 5.53 | 0.46× |
| high-frequency | 100,000 | 11.89 | 4.77 | 0.40× |
| balanced | 1,000 | 16.81 | 8.15 | 0.49× |
| balanced | 10,000 | 15.77 | 5.22 | 0.33× |
| balanced | 100,000 | 15.35 | 4.78 | 0.31× |

Paid for in encode CPU and allocations. At n=10k zstd, Arrow p50 is 15.6 ms
vs proto's 2.58 ms (6×); at n=100k it's 209 ms vs 23 ms (9×). Reach for
Arrow when bandwidth is the binding constraint, microbatches reach ≥ 1,000
datapoints, and the producer can afford the CPU.

### Arrow is not safe for high-cardinality metrics

On the high-cardinality profile, Arrow shows **no size advantage at any
batch size** and the upstream producer **panics at n = 100,000** ("Too many
consecutive schema updates"). At n=10k zstd, Arrow `B/rec` is 38.1 vs
proto's 37.3 — equal on size, ~7× slower to encode. For workloads where
each datapoint carries a unique value (request id, user id, trace fragment),
pick proto or JSON; Arrow's per-record schema model fights the shape.

### Codec ranking

Within an encoding, codec choice has a clear ranking that holds across
profiles and batch sizes:

1. **`zstd`** — best ratio per CPU cycle. Default.
2. **`snappy` / `x-snappy-framed`** — half the ratio of zstd but 2–3× faster encode. Under CPU pressure.
3. **`gzip` / `zlib` / `deflate`** — equivalent ratios to each other, slightly better than zstd in a few cells, but 3–6× slower than zstd. Only for downstream compatibility.
4. **`none`** — only at n=1, where compression overhead exceeds the bytes saved.

### Picking a batch size

`B/rec` falls steeply from n=1 to n=1000 and plateaus thereafter for proto
and JSON. Arrow keeps falling all the way to n=100,000. For most
deployments **n ≈ 1,000 is the knee** — diminishing returns on batching
beyond that for proto/JSON, and a reasonable inflection point if you've
chosen Arrow.

### Pair with the Kinesis record ceiling

`total` tells you when a configuration would exceed a stream's per-record
limit. With `max_record_size: 1 MiB` (default), at high-frequency zstd:

- proto/JSON safely batch to n=100,000 (~1 MiB record)
- Arrow has ample headroom (n=100k → 466 KiB)

With `max_record_size: 10 MiB` (the modern Kinesis ceiling, opt-in), the
size question stops being "will it fit" and becomes "what CPU and tail
latency am I willing to spend."
